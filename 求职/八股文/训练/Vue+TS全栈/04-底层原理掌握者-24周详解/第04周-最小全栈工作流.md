# 第04周 最小全栈工作流

## 本周你到底在学什么

前三周学的是零件，这一周学的是把零件装起来，形成“请求进来 -> 处理 -> 返回 -> 展示”的最小闭环。

## 先把原理讲透

- 全栈协作的关键不是“前后端都能跑”，而是边界一致。
- 一条链路里至少有四件事：输入、验证、业务处理、输出。
- 只要这四件事混在一起，项目就会很快失控。

## 最小例子

```ts
type Todo = { id: number; title: string };

function handleCreateTodo(input: { title?: string }) {
  if (!input.title) {
    return { ok: false as const, error: "title 不能为空" };
  }
  const todo: Todo = { id: Date.now(), title: input.title };
  return { ok: true as const, data: todo };
}
```

一个最小 handler 都已经包含了输入校验、业务处理、返回建模三层含义。

## 放到 Vue3 / TS / Node.js 里

- Vue3 负责把状态和交互展示给用户。
- TypeScript 负责把“传什么、回什么、错什么”说清楚。
- Node.js 负责把请求接进来，并把业务结果转成 HTTP 语义。

## 实际应用场景

- Todo 新增流程。
- 登录表单提交流程。
- 文件上传前的参数校验流程。

## 常见误区

- 误区：先把接口跑通，类型和错误以后再补。
  - 这样后面返工最痛。
- 误区：全栈闭环就是“前端请求后端成功了”。
  - 其实失败分支、边界输入、状态回显同样重要。

## 本周实践

- 自己写一条最小业务链路，从输入到展示都带上错误分支。
- 画一张图，把每一步职责拆开。
- 用口头语言解释：为什么“工作流”比“单点功能”更值得优先学。

## 学完你应该能说清

- 一条最小全栈链路应该包含哪些层。
- 为什么参数校验和业务处理不能混成一个大函数。
- 为什么前端状态设计其实依赖后端返回模型。
