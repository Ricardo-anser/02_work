# 第07周 组合式 API 与状态边界

## 本周你到底在学什么

这一周要解决的问题是：状态应该放哪，副作用应该放哪，可复用逻辑应该怎么抽。

## 先把原理讲透

- 组合式 API 的核心不是“把代码拆出去”，而是“按逻辑关注点组织代码”。
- 状态边界的核心不是“是不是全局”，而是“谁拥有它、谁负责修改它、谁依赖它”。
- 一旦页面局部态、业务共享态、用户态全混一起，项目很快就会发臭。

## 最小例子

```ts
import { ref } from "vue";

export function useCounter() {
  const count = ref(0);
  const inc = () => {
    count.value += 1;
  };
  return { count, inc };
}
```

这个例子看起来简单，但已经体现了组合式 API 的价值：把“状态 + 操作”封成一块。

## 放到 Vue3 / TS / Node.js 里

- Vue3 的 composable 是逻辑复用单元，不是简单工具函数复制。
- TypeScript 让 composable 的输入输出边界更清晰。
- Node.js 虽然没有 composable，但 service、middleware、helper 的边界思维是类似的。

## 实际应用场景

- 列表搜索条件抽成 `useSearch`。
- 分页逻辑抽成 `usePagination`。
- 登录态、权限态、页面局部弹窗状态分层管理。

## 常见误区

- 误区：能抽出去的都叫 composable。
  - 如果没有明确输入输出和依赖边界，抽出去只会更乱。
- 误区：状态只分“本地”和“全局”。
  - 实际至少要区分局部态、业务共享态、用户态、服务端态。

## 本周实践

- 选一个页面，把重复逻辑抽成两个 composable。
- 画出该页面的状态分层图。
- 用一句话解释“为什么状态边界比 API 数量更重要”。

## 学完你应该能说清

- composable 为什么是“逻辑模块”，不是“语法糖”。
- 什么状态不应该进 store。
- 为什么副作用和纯状态最好分开放。
