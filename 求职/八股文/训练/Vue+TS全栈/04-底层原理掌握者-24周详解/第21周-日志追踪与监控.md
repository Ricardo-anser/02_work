# 第21周 日志、追踪与监控

## 本周你到底在学什么

这周要理解“系统为什么必须可观测”，而不是等出问题时只会翻控制台。

## 先把原理讲透

- 日志回答“发生了什么”。
- 指标回答“整体趋势如何”。
- 追踪回答“一次请求到底经过了哪几步”。
- 三者不是互相替代，而是互相补位。

## 最小例子

```js
console.log(JSON.stringify({
  level: "info",
  requestId: "req-1",
  message: "create article start"
}));
```

哪怕只是一个结构化日志，也比一行随手 `console.log("开始了")` 有价值得多。

## 放到 Vue3 / TS / Node.js 里

- Vue 前端需要错误边界、关键用户行为、请求失败上下文。
- TS 让日志字段结构更稳定。
- Node 要负责 request-id、耗时、错误、依赖状态的统一记录。

## 实际应用场景

- 用户说“保存失败”，你要知道失败在哪一层。
- 接口偶尔慢，要知道是数据库慢、缓存未命中还是第三方依赖慢。
- 某条链路只在生产出错，日志和追踪是第一手证据。

## 常见误区

- 误区：能看到报错栈就够了。
  - 真实系统里，很多问题靠单个报错根本拼不出全貌。
- 误区：日志越多越好。
  - 没结构、没层次、没关联 ID 的日志只会制造噪音。

## 本周实践

- 设计一套最小结构化日志字段。
- 解释 request-id 为什么重要。
- 用 3 个案例区分日志、指标、追踪分别最适合回答什么问题。

## 学完你应该能说清

- 可观测性为什么是系统设计的一部分。
- 为什么“出问题了再加日志”通常太晚。
- 日志、指标、追踪三者各自最擅长什么。
