# 第18周 SSR、CSR 与 Hydration

## 本周你到底在学什么

这周要理解“同一份页面为什么会有服务端先产 HTML、客户端再接管”的模式，以及它最容易出什么问题。

## 先把原理讲透

- CSR 是浏览器拿到 JS 后自己生成界面。
- SSR 是服务器先把首屏 HTML 算出来再发给浏览器。
- Hydration 解决的是“客户端接手服务器已经画好的页面，而不是推倒重来”。
- 所以最关键的要求是：两边第一次渲染结果必须尽量一致。

## 最小例子

```js
const value = Math.random();
```

为什么这一行在 SSR 页面里很危险？因为服务端渲染一次、客户端再渲染一次，结果很可能不同，从而造成 hydration mismatch。

## 放到 Vue3 / TS / Node.js 里

- Vue / Nuxt 的 SSR 重点是“数据预取 + 首屏 HTML + 客户端接管”。
- TS 在这里帮助你约束首屏数据结构和传输边界。
- Node 负责服务端渲染时的数据准备与环境隔离。

## 实际应用场景

- 博客详情页。
- 面向搜索引擎的营销页。
- 需要更快首屏体验的内容系统。

## 常见误区

- 误区：SSR 只是为了 SEO。
  - 首屏体验、内容可见性、弱网可用性都相关。
- 误区：同构就是前后端代码完全一样。
  - 真正难点是环境差异和状态接管。
