# 第02周 异步流程与错误边界

## 本周你到底在学什么

你不是在背 Promise API，而是在学“结果为什么不会立刻回来”和“出错时它究竟沿哪条路往外冒”。

## 先把原理讲透

- 同步代码的特点是：写在下一行，就在上一行后立刻执行。
- 异步代码的特点是：你先登记一个未来动作，真正执行要等时机成熟。
- Promise 解决的是“异步结果如何被有组织地接住”。
- 错误边界解决的是“哪一层负责兜底，哪一层只负责往上抛”。

## 最小例子

```js
function getUser() {
  return Promise.resolve({ id: 1, name: "Ada" });
}

getUser()
  .then((user) => {
    throw new Error(`用户 ${user.name} 加载后出错`);
  })
  .catch((error) => {
    console.error(error.message);
  });
```

这个例子最值得看的不是成功分支，而是：`then` 里抛的错，为什么 `catch` 能接住。

## 放到 Vue3 / TS / Node.js 里

- Vue3 里页面 loading、error、success 三态，本质是 Promise 状态外显。
- TypeScript 里如果错误类型不建模，前端就只能用 `any` 胡猜。
- Node.js 里如果每层都 try/catch，又不重新抛错，很快就会出现“错误消失了”的排障噩梦。

## 实际应用场景

- 表单提交失败，是提示用户重试、静默吞掉、还是降级展示？
- 请求超时，是页面级错误、组件级错误，还是全局错误？
- 后端 service 抛错后，是 route 层转 HTTP 响应，还是底层直接决定状态码？

## 常见误区

- 误区：异步错误就是网络错误。
  - 实际上还有代码抛错、解析失败、业务失败三类。
- 误区：用了 `async/await` 就没有 Promise 了。
  - 其实只是写法变了，底层仍然是 Promise 链。

## 本周实践

- 写一个 `retry(fn, count)` 函数，失败重试两次。
- 分别用 `.then()` 和 `async/await` 实现。
- 画一张图说明错误从最底层函数一路冒到页面提示的路线。

## 学完你应该能说清

- Promise 为什么能把异步流程“串起来”。
- `throw` 和 `reject` 在链路上为什么表现类似。
- 一个系统里，错误应该在哪一层被真正处理掉。
