# 第15周 HTTP 服务与错误模型

## 本周你到底在学什么

你这周学的不是“怎么起一个服务”，而是“一个请求进来后，系统应该在哪些层做什么，以及出错时怎么统一收口”。

## 先把原理讲透

- HTTP 服务至少包含：路由、参数解析、业务处理、响应输出。
- 错误模型至少要回答：谁抛错、谁转换、谁记录、谁提示。
- 中间件的价值在于把共性动作放到统一位置，而不是散落在每个 handler 里。

## 最小例子

```js
async function handler(req, res) {
  try {
    const data = await service();
    res.end(JSON.stringify({ ok: true, data }));
  } catch (error) {
    res.statusCode = 500;
    res.end(JSON.stringify({ ok: false, message: error.message }));
  }
}
```

这个例子简单，但已经揭示了一个关键问题：业务错和 HTTP 响应错，不是同一层概念。

## 放到 Vue3 / TS / Node.js 里

- Vue 前端只关心“结果长什么样”，但这个结构稳定与否取决于后端错误模型。
- TS 让错误响应结构可预测，不至于前端到处 `error?.message ??` 猜。
- Node 里中间件和 handler 的边界清楚，接口层才不会越写越粘。

## 实际应用场景

- 登录失败返回 401。
- 资源不存在返回 404。
- 参数非法返回 422。

## 常见误区

- 误区：后端报错了就统一 500。
  - 这会让前端和用户完全分不清问题性质。
- 误区：每个 handler 自己随便返回结构。
  - 这样联调会非常痛苦。

## 本周实践

- 设计一个统一响应结构。
- 设计 3 类不同错误并给出 HTTP 语义。
- 解释为什么“错误码设计”本质上是系统沟通能力的一部分。

## 学完你应该能说清

- HTTP 语义和业务语义的关系。
- 中间件为什么能减少重复逻辑。
- 为什么稳定错误模型能显著提升前后端协作效率。
