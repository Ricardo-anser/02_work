# 第12周 类型驱动工具

## 本周你到底在学什么

这周要把前面学的 TS 能力从“会写类型”推进到“能做一个小工具服务项目”。

## 先把原理讲透

- 工具化的意义在于把重复判断、重复拼接、重复映射收束起来。
- 类型驱动工具并不一定很大，它可能只是一个 typed client、一个表单辅助器、一个事件类型生成器。
- 重点不是代码量，而是“类型信息真的流动起来了”。

## 最小例子

```ts
function createGetter<T extends object>(obj: T) {
  return function <K extends keyof T>(key: K): T[K] {
    return obj[key];
  };
}
```

这个小工具的价值在于：调用方拿到的不是模糊的值，而是和 key 对应的精确类型。

## 放到 Vue3 / TS / Node.js 里

- Vue3 项目里可做 typed fetch、typed form helper。
- Node.js 项目里可做统一的 API result builder。
- TS 的价值在这里最明显：同一份工具，给多个模块提供一致边界。

## 实际应用场景

- 请求结果统一包装。
- 事件 payload 自动约束。
- 表单字段 getter/setter 自动推断。

## 常见误区

- 误区：工具越通用越好。
  - 过度通用通常意味着复杂、难读、难维护。
- 误区：只要用了泛型就叫类型驱动。
  - 关键是它是否真的减少重复和出错点。

## 本周实践

- 选一个真实项目痛点做小工具。
- 限制范围：只解决 1 个核心问题，不贪大。
- 写 2 个成功例子和 1 个失败例子证明它值不值得保留。

## 学完你应该能说清

- 工具化和抽象的边界在哪里。
- 为什么好的工具应该先服务当前项目，再谈通用化。
- 类型驱动工具如何减少重复劳动。
