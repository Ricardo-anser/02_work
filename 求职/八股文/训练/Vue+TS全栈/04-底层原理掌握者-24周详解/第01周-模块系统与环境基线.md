# 第01周 模块系统与环境基线

## 本周你到底在学什么

这一周不是背 `import` 和 `require` 的语法，而是要建立一个最重要的感觉：代码不是“放在文件里就能互相看到”，而是必须通过模块系统建立边界、暴露接口、决定加载时机。

## 先把原理讲透

- 模块的本质是“把一段代码的作用域封起来，再决定向外暴露什么”。
- 浏览器原本只认识脚本标签，Node.js 最早用 CommonJS，后来才支持 ESM。
- 所以你现在看到的 `import/export` 和 `require/module.exports`，不是两个写法，而是两套设计哲学。
- CommonJS 更像“运行时把文件执行一遍，再拿执行结果”。
- ESM 更像“在执行前先建立依赖图，再按图装配模块关系”。

## 最小例子

```js
// math.mjs
export function add(a, b) {
  return a + b;
}

// index.mjs
import { add } from "./math.mjs";
console.log(add(1, 2));
```

先问自己两个问题：

1. `add` 是什么时候被发现的，是执行后才知道，还是解析时就知道？
2. 如果路径写错，报错发生在什么时候？

## 放到 Vue3 / TS / Node.js 里

- Vue3 里的异步组件，本质上也依赖 `import()` 的模块加载语义。
- TypeScript 在写类型检查前，首先也要完成模块解析，不然连“这个标识符来自哪里”都不知道。
- Node.js 里你之所以总会遇到 `ERR_REQUIRE_ESM`，就是因为模块系统边界没对齐。

## 实际应用场景

- 拆前端项目时，页面、组件、hooks、store 都依赖模块边界。
- 拆后端项目时，route、service、repo、shared types 也依赖模块边界。
- 团队协作里，模块边界清晰，代码 review 才会快；不然任何文件都能改任何东西。

## 常见误区

- 误区：`import` 只是比 `require` 新一点。
  - 实际上它影响静态分析、tree-shaking、打包优化和循环依赖表现。
- 误区：写前端就不用懂 CommonJS。
  - 实际上脚手架、工具链、旧包兼容都可能碰到它。

## 本周实践

- 自己写一份 ESM demo 和一份 CJS demo，做同样功能。
- 故意把扩展名、省略路径、错误导入方式都试一遍。
- 用一句通俗话解释：为什么“模块系统”其实是项目结构的地基。

## 学完你应该能说清

- 为什么 ESM 更适合静态分析。
- 为什么 Node.js 里不能随便混用两套模块系统。
- 为什么“项目拆模块”不是整理文件夹，而是在设计依赖关系。
