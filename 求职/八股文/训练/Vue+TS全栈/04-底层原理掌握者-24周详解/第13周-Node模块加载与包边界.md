# 第13周 Node 模块加载与包边界

## 本周你到底在学什么

这周要从“会装包”升级到“理解一个包是怎么被 Node 找到、加载、暴露出来的”。

## 先把原理讲透

- 文件是文件，包是包，这两层不要混。
- 模块解析解决的是“去哪里找”，模块加载解决的是“找到后怎么执行”。
- `exports`、`main`、`type`、扩展名这些配置，本质都在控制包边界。

## 最小例子

```json
{
  "name": "demo-pkg",
  "type": "module",
  "exports": {
    ".": "./index.js",
    "./utils": "./utils.js"
  }
}
```

这个配置的重点不是记字段，而是理解：作者正在决定“外部只能走哪些入口进来”。

## 放到 Vue3 / TS / Node.js 里

- Vue 工程里的组件库、hooks 包、shared 包都依赖包边界。
- TS 类型声明是否能被正确引用，也依赖导出结构。
- Node 后端分层后，入口混乱通常就是包边界设计差。

## 实际应用场景

- monorepo 中的 shared 包。
- 内部工具库只公开稳定 API，不暴露内部实现。
- 后端模块逐渐演化为可复用包。

## 常见误区

- 误区：只要文件能 import 到就没问题。
  - 深路径依赖一旦被外部绑住，重构会很痛。
- 误区：包边界只是发 npm 包时才要关心。
  - 大型项目内部模块同样需要它。

## 本周实践

- 设计一个只暴露 2 个入口的小包。
- 解释 `exports` 为什么比“到处深路径 import”更稳。
- 画出你的项目里哪些目录其实已经具备“包”的形状。

## 学完你应该能说清

- 模块解析和模块加载分别处理什么问题。
- 为什么包边界是可维护性的基础设施。
- 为什么“能引用到”不等于“应该引用到”。
