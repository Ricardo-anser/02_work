# 第06周 Vue 渲染与更新

## 本周你到底在学什么

响应式只是“知道该更新了”，这一周学的是“更新到底怎么发生，以及为什么不会每改一次就暴力整页重画”。

## 先把原理讲透

- 渲染的本质是“状态 -> 视图表示”。
- 更新的本质是“新状态对应的新视图，与旧视图做对比，再决定最小改动”。
- 调度的本质是“别急着立刻更新，先把同一轮修改合并起来”。

## 最小例子

```js
const queue = new Set();

function schedule(job) {
  queue.add(job);
  Promise.resolve().then(() => {
    for (const fn of queue) fn();
    queue.clear();
  });
}
```

这就是“多次状态变更合并成一轮刷新”的最小调度直觉。

## 放到 Vue3 / TS / Node.js 里

- Vue3 的批量更新、nextTick，本质都离不开调度。
- TypeScript 在这里主要帮助你维护 vnode、props、组件入参这些结构。
- Node.js 不做 DOM 渲染，但它的事件循环与任务调度思路能帮助你理解“为什么更新不是同步一锤子下去”。

## 实际应用场景

- 一个输入框每敲一个字都触发多处状态更新，但页面不会抖得失控。
- 列表筛选、排序、分页一起改时，最终合并成一轮视图刷新。
- 复杂表单里多个字段联动时，不会每个字段都强制重渲染全页面。

## 常见误区

- 误区：响应式触发了，就一定立刻重渲染。
  - 实际上中间经常有调度层。
- 误区：组件更新慢，一定是 diff 算法慢。
  - 很多时候是状态边界太乱，导致无效更新太多。

## 本周实践

- 写一个最小 scheduler，把连续 5 次修改合并成 1 次输出。
- 画一张图说明“状态变更 -> 调度 -> 渲染”三层。
- 找一个现成 Vue 页面，分析它可能有哪些无效更新。

## 学完你应该能说清

- 调度为什么是渲染系统的关键一层。
- 为什么“最小更新”比“整树重画”更适合复杂应用。
- 页面卡顿时，为什么要先查状态边界，再查渲染细节。
