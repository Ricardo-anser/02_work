# 第11周 类型共享与接口对齐

## 本周你到底在学什么

这一周要解决一个非常现实的问题：前端和后端说的是不是同一种“语言”。

## 先把原理讲透

- 共享类型不是把后端所有模型都丢给前端。
- 接口对齐的重点不是“有没有同名字段”，而是“语义是否一致”。
- 真正应该共享的是传输契约，而不是数据库内部结构。

## 最小例子

```ts
export type LoginRequest = {
  username: string;
  password: string;
};

export type LoginResponse = {
  token: string;
  userName: string;
};
```

简单，但它已经体现了共享契约的价值：前后端都围着这份协议写代码。

## 放到 Vue3 / TS / Node.js 里

- Vue3 组件和页面可以直接消费共享的 response 类型。
- Node.js handler 和 service 用同一份 request/response 类型，减少漂移。
- TS 在这里真正承担起“跨层契约语言”的角色。

## 实际应用场景

- 登录接口。
- 分页列表接口。
- 内容发布与编辑接口。

## 常见误区

- 误区：ORM 模型就是共享类型。
  - 数据库存储形态和前端展示形态往往不同。
- 误区：共享类型越多越好。
  - 共享错了边界，会让前后端一起被内部结构绑死。

## 本周实践

- 设计一份 `auth` 相关共享类型。
- 用一句话解释 DTO 和数据库模型的差别。
- 找出你项目里最容易“前后端各写一份”的字段结构。

## 学完你应该能说清

- 什么类型该共享，什么不该共享。
- 为什么共享类型解决的是协作一致性问题。
- 为什么“字段一样”不等于“语义一致”。
