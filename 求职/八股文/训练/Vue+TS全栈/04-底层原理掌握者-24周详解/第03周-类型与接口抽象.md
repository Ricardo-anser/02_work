# 第03周 类型与接口抽象

## 本周你到底在学什么

这周不是学“怎么写 interface”，而是学“怎么用类型把输入、输出、失败分支讲清楚”。

## 先把原理讲透

- 类型系统最大的价值不是提示更智能，而是让你在代码运行前先发现语义冲突。
- 接口抽象的重点不是“字段列全”，而是“边界说清”。
- 前端和后端最容易出问题的地方，往往不是算法，而是“我以为这个字段一定有，你以为那个字段可能没有”。

## 最小例子

```ts
type ApiResult<T> =
  | { ok: true; data: T }
  | { ok: false; error: string };

function printUser(result: ApiResult<{ name: string }>) {
  if (!result.ok) {
    console.error(result.error);
    return;
  }
  console.log(result.data.name);
}
```

这个例子里的关键，不是语法，而是“成功和失败都被建模了”。

## 放到 Vue3 / TS / Node.js 里

- Vue3 页面状态其实很适合用可辨识联合表达。
- TypeScript 的类型约束让你写请求层时不再靠猜字段。
- Node.js 的 handler 如果有统一的入参与出参模型，联调会轻松很多。

## 实际应用场景

- 登录接口可能成功返回 token，也可能失败返回错误码。
- 列表接口可能返回分页信息，详情接口则不需要。
- 表单提交时，字段错误和全局错误不是一回事。

## 常见误区

- 误区：后端返回什么，前端就直接照单全收。
  - 实际上应该先有契约，再有实现。
- 误区：一个 `User` 类型走天下。
  - 实际上创建用户、返回用户、展示用户通常是三种视图。

## 本周实践

- 为“用户登录”设计请求、成功响应、失败响应三类类型。
- 把“分页列表返回值”抽成泛型容器。
- 用 3 句话解释：为什么“类型系统”本质上是在约束协作边界。

## 学完你应该能说清

- 什么叫“把失败也建进类型里”。
- DTO、领域模型、数据库模型为什么不该混成一个。
- 类型抽象为什么能减少联调成本。
