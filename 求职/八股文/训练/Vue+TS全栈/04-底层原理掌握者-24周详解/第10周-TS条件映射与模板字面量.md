# 第10周 TS 条件、映射与模板字面量

## 本周你到底在学什么

这一周开始进入 TS 最有“元编程味道”的部分：让类型也能做变换，而不是只当静态标签。

## 先把原理讲透

- 条件类型是在类型层做 if/else。
- 映射类型是在类型层做“批量加工字段”。
- 模板字面量类型是在类型层拼接字符串结构。
- 三者结合，能把大量重复定义自动化掉。

## 最小例子

```ts
type FormErrors<T> = {
  [K in keyof T]?: string;
};

type LoginForm = {
  username: string;
  password: string;
};

type LoginErrors = FormErrors<LoginForm>;
```

重点不是写法，而是你从“手写两个错误字段”变成了“让类型跟表单结构一起变”。

## 放到 Vue3 / TS / Node.js 里

- Vue3 的表单错误模型、事件名映射、组件 prop 转换都很适合。
- Node.js 的路由名、事件总线、响应包装同样适合模板字面量类型。
- 这类能力的价值在中大型项目里才会越来越明显。

## 实际应用场景

- 表单字段错误映射。
- `user:create`、`user:update` 这类事件名约束。
- API 响应去掉只读、改可选、批量包装字段。

## 常见误区

- 误区：高级类型越复杂越高级。
  - 真正好的类型应该“减少重复，同时不伤可读性”。
- 误区：类型体操只适合刷题。
  - 实际表单、事件、请求层都很常用。

## 本周实践

- 为一个表单写 `FormErrors<T>`。
- 为一个事件系统写模板字面量事件名。
- 把你写过的 1 组重复类型改造成映射类型。

## 学完你应该能说清

- 条件类型、映射类型、模板字面量分别擅长解决什么问题。
- 什么时候应该为了减少重复上高级类型，什么时候不值得。
- 类型层自动化为什么本质上是在维护一致性。
