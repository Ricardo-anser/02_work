# 第13周 Node 模块加载与包边界

## 这周为什么重要

项目进入后端重整阶段后，你会发现很多问题不是业务错，而是目录边界和模块依赖太乱。

## 先讲明白原理

- route、service、repo、shared 不是为了分层而分层，而是为了让改动影响面变小。
- 包边界不清，后面重构会经常牵一发而动全身。
- 交付系统时，结构清楚比“先把功能堆上去”更能保命。

## 最小例子

```txt
src/
  routes/
  services/
  repositories/
  shared/
```

这不是模板，而是在回答：请求入口在哪、业务逻辑在哪、数据访问在哪、公共能力在哪。

## 放进真实项目怎么理解

- Vue 项目会有页面、组件、composable、store 的边界。
- Node 项目则需要 route、service、repo 的边界。
- 两边都在解决同一件事：降低耦合。

## 实际应用场景

- 后端 API 模块持续增长。
- 需要多人同时改不同业务。
- 准备把公共逻辑抽出来复用。

## 容易踩的坑

- route 里直接写数据库逻辑。
- 配置读取到处散落。
- 任何目录都能反向 import 任意目录。
