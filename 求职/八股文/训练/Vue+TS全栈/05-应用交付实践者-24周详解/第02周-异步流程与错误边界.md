# 第02周 异步流程与错误边界

## 这周为什么重要

真实项目里，用户最容易感知的问题不是“算法写得优不优雅”，而是“按钮点了没反应”“报错了不知道去哪看”“失败后页面卡死”。

## 先讲明白原理

- 一次请求至少会经历：发出、等待、成功或失败、页面反馈。
- 如果你不提前设计 loading/error/success 三态，页面体验一定会乱。
- 错误边界不清，前端和后端就会互相甩锅。

## 最小例子

```ts
type RequestState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; message: string };
```

这个模型的价值是：页面不再靠一堆零散布尔值猜自己现在处于什么状态。

## 放进真实项目怎么理解

- Vue 页面需要稳定展示 loading、错误、重试按钮。
- Node 接口需要稳定返回错误结构。
- 类型系统要让这两者在联调时说同一种语言。

## 实际应用场景

- 登录失败提示。
- 列表加载失败重试。
- 提交表单后按钮禁用与恢复。

## 容易踩的坑

- 只有成功态，没有失败态。
- 请求报错后页面不恢复，按钮一直 loading。
- 后端错误结构每个接口都不一样。

- 做一个带 loading/error/success 的页面交互。
