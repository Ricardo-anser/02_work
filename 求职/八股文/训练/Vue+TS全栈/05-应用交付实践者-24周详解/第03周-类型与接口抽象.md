# 第03周 类型与接口抽象

## 这周为什么重要

应用交付里最常见的返工，往往不是功能做错，而是接口理解不一致。第三周就是专门拿掉这个雷。

## 先讲明白原理

- 一个接口至少要说清 3 件事：传什么、回什么、失败长什么样。
- 页面状态、表单结构、后端 DTO 如果没有统一语言，项目复杂度会指数上涨。
- 类型抽象不是为了学术，而是为了减少沟通成本。

## 最小例子

```ts
type CreateTodoRequest = {
  title: string;
};

type CreateTodoResponse = {
  id: number;
  title: string;
  completed: boolean;
};
```

简单，但它已经比“后端先写，前端看着猜”稳定得多。

## 放进真实项目怎么理解

- Vue 表单字段要和接口入参对齐。
- Node handler 出参要和页面使用方式对齐。
- 共享类型要选“传输视图”，不是数据库表结构。

## 实际应用场景

- 登录接口。
- 列表分页接口。
- 内容创建与编辑接口。

## 容易踩的坑

- 一个 `User` 类型从数据库用到页面。
- 错误响应没有统一字段。
- 前端自己多加或少用字段，直到联调才发现。

## 本周交付

- 为 1 个业务模块写清 request/response/error 契约。
- 让前后端共享这份契约。
- 说清楚：为什么“接口抽象”其实是交付速度加速器。
