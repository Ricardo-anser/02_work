# 第01周 模块系统与环境基线

## 这周为什么重要

很多人一上来就开始写页面和接口，结果第二周目录就乱了、第三周环境就炸了。应用交付路线的第一周，真正目标是先把仓库骨架和运行上下文立住。

## 先讲明白原理

- 项目结构不是装饰，而是在提前决定协作边界。
- 前端、后端、共享类型如果一开始就混在一起，后面所有交付动作都会变慢。
- 模块模式选不清，后面 `import`、构建、脚本执行都会出问题。

## 最小例子

```txt
apps/
  web/
  api/
packages/
  shared-types/
```

这个结构看似普通，但它已经回答了 3 个交付问题：谁负责页面，谁负责接口，谁负责共享契约。

## 放进真实项目怎么理解

- Vue3 前端是一个独立运行单元。
- Node API 是另一个独立运行单元。
- shared types 是连接两者的协作桥梁，而不是杂物间。

## 实际应用场景

- 做 Todo 系统时，前端和后端同步推进。
- 做后台项目时，登录、列表、详情页都依赖统一类型出口。
- 做多人协作时，目录边界越清楚，扯皮越少。

## 容易踩的坑

- 一开始只建一个 `src`，什么都往里塞。
- 前后端都手写同一份类型。
- 本地能跑，但没人说得清每个命令在哪个目录执行。

## 本周交付

- 建好前端、后端、共享类型三块骨架。
- 记录启动命令和执行上下文。
- 用一句话说清：为什么“先搭结构”不是浪费时间。
