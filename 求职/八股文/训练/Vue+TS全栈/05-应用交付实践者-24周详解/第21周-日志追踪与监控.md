# 第21周 日志、追踪与监控

## 这周为什么重要

当项目已经能跑，下一步最现实的问题就是：线上一出错，你凭什么快速定位。

## 先讲明白原理

- 日志让你知道单次行为发生了什么。
- 指标让你知道整体是不是在变坏。
- 追踪让你知道一条请求到底卡在哪。
- 可观测性不是运维专属，而是应用交付成熟度的一部分。

## 最小例子

```json
{
  "requestId": "req-42",
  "path": "/api/articles",
  "duration": 135
}
```

一条结构化日志就已经比散乱 console 输出更接近真实生产世界。

## 放进真实项目怎么理解

- 前端要记录关键失败和错误边界。
- 后端要记录请求链路和核心依赖耗时。
- 监控体系会直接影响你排障速度。

## 实际应用场景

- 保存文章失败。
- 登录接口偶发超时。
- 缓存层异常导致响应抖动。

## 容易踩的坑

- 只有错误栈，没有上下文。
- 日志没有 request-id。
- 明明线上慢，却没人知道慢在哪一层。
