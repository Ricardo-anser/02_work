# 第07周 组合式 API 与状态边界

## 这周为什么重要

当页面数量变多，你会明显感觉：问题不再是不会写，而是哪里都能写，结果哪里都难维护。

## 先讲明白原理

- composable 解决的是逻辑复用。
- store 解决的是共享状态协作。
- 页面局部态解决的是当前页面自己的交互细节。
- 这三者不分清，项目就会很快进入“改一处坏三处”。

## 最小例子

```ts
export function usePagination() {
  const page = ref(1);
  const pageSize = ref(10);
  return { page, pageSize };
}
```

这个例子很小，但已经体现了“把一个逻辑关注点独立成单元”的价值。

## 放进真实项目怎么理解

- 搜索、分页、排序适合做 composable。
- 当前用户、权限、主题这类适合进 store。
- 弹窗开关、输入框值通常只属于局部态。

## 实际应用场景

- 用户列表页。
- CMS 编辑页。
- 登录与权限控制。

## 容易踩的坑

- 只要能共享就塞到 store。
- composable 没有明确输入输出。
- 页面局部态被全局状态污染。
