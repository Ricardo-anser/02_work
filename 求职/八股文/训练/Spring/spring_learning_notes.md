# Spring 学习笔记

## 1. 学习目标

这份笔记试图回答 4 个核心问题：

- 注解到底是什么
- 注解为什么能生效
- `@Aspect` 和注解是什么关系
- Spring 应该如何建立整体心智模型

如果这 4 个问题打通，Spring 就不会再像“注解黑魔法”，而会变成一套可以拆开的系统：

- 容器
- Bean 定义
- 生命周期
- 元数据解释
- 代理与增强
- 上层子系统

---

## 2. 注解的本质

### 2.1 注解是什么

注解本质上是 `元数据（metadata）`。

它不是业务逻辑本身，而是写给这些“解释者”看的结构化信息：

- 编译器
- IDE
- 框架
- 运行时反射机制
- 编译期处理器

一句话理解：

`注解 = 写给解释器看的声明`

### 2.2 注解不是什么

注解不是：

- 业务逻辑本身
- 方法增强本身
- 自动执行器
- 魔法开关

它自己不会生效，必须有人“读取它并采取行动”。

---

## 3. 注解为什么会生效

### 3.1 生效链条

任何注解生效，本质都要经过这条链：

- 定义注解
- 把注解标在类 / 方法 / 字段 / 参数上
- 有某个机制读取注解
- 读取后执行对应逻辑

所以统一模型是：

- 注解：声明
- 反射 / 扫描 / 编译器：发现
- 代理 / 调度器 / 处理器：执行

### 3.2 常见解释器

不同注解由不同解释器负责：

- `@Override`
  - 编译器解释
- `@Deprecated`
  - 编译器和 IDE 解释
- `@Autowired`
  - Spring 容器解释
- `@Transactional`
  - Spring AOP / 事务拦截器解释
- `@RequestMapping`
  - Spring MVC 映射器解释

结论：

`注解自己不做事，真正做事的是解释它的那套机制`

---

## 4. 注解的关键元注解

### 4.1 `@Target`

指定注解能标在哪些程序元素上：

- `TYPE`
- `METHOD`
- `FIELD`
- `PARAMETER`

### 4.2 `@Retention`

指定注解保留到什么时候：

- `SOURCE`
  - 只在源码里存在
  - 编译后丢弃
- `CLASS`
  - 编译后进入 `.class`
  - 运行时默认拿不到
- `RUNTIME`
  - 运行时还能通过反射拿到

最重要结论：

`Spring / 反射 / AOP 想在运行时读取注解，通常必须是 RUNTIME`

### 4.3 `@Documented`

让注解进入 javadoc。

### 4.4 `@Inherited`

让子类继承父类上的类级注解。

注意：

- 它只对 `类级注解` 有意义
- 对方法注解没用

---

## 5. 自定义注解的三种实现思路

### 5.1 运行时反射版

模式：

- 自定义注解
- 运行时反射读取
- 手动调用器执行逻辑

特点：

- 最适合理解原理
- 最直接暴露“注解不会自己生效”

模型：

`注解 + 反射 + 显式调度`

适合：

- 教学
- 小型工具
- 最小 demo

### 5.2 动态代理版

模式：

- 方法打注解
- 通过代理拦截方法调用
- 调用前后做增强

特点：

- 更接近 Spring AOP
- 用户调用时更自然

模型：

`注解 + 代理 + 隐式增强`

适合：

- 权限
- 日志
- 审计
- 缓存
- 事务

### 5.3 APT 编译期版

模式：

- 注解只在编译期存在
- 编译器处理器扫描注解
- 自动生成代码或注册表

特点：

- 不依赖运行时反射
- 错误能更早暴露
- 构建链更复杂

模型：

`注解 + 编译期处理器 + 代码生成`

适合：

- 自动注册表
- 自动工厂
- 自动路由表
- 代码生成框架

## 6. Spring 中的自定义注解

Spring 里的自定义注解，常见有 3 条路线。

### 6.1 组合注解

例子：

- `@BizService`
- `@AdminController`

本质：

- 用 Spring 现有注解做二次包装
- 提高业务语义
- 统一团队风格

常和这些搭配：

- `@Service`
- `@Component`
- `@RestController`
- `@AliasFor`

核心思想：

`不是新增能力，而是重新组织已有能力`

### 6.2 业务注解 + AOP

例子：

- `@Audit`
- `@PermissionCheck`
- `@RateLimit`

本质：

- 业务注解声明需求
- 切面拦截并执行横切逻辑

核心思想：

`声明式横切编程`

### 6.3 业务注解 + Spring 扩展点

典型扩展点：

- `BeanPostProcessor`
- `BeanFactoryPostProcessor`
- `ImportSelector`
- `HandlerMethodArgumentResolver`

这种方式更像“写框架”，不是普通业务层最常用的玩法。

## 7. `@Aspect` 和注解的关系

### 7.1 `@Aspect` 本身也是注解

它的作用是：

- 告诉 Spring：这是一个切面类

所以：

`@Aspect` 是声明“切面身份”的注解`

### 7.2 Aspect 经常拿别的注解作为切入条件

例如：

- 自定义 `@Audit`
- 切面里用 `@annotation(Audit)` 匹配方法

关系变成：

- 业务注解：表达“这里需要某种横切能力”
- 切面：表达“匹配到后做什么”

所以：

`业务注解负责说“我要什么”，Aspect 负责说“怎么处理”`

### 7.3 真正把二者串起来的是代理

Spring AOP 的常见链路：

- Spring 发现切面
- 解析切点表达式
- 为目标 Bean 创建代理
- 调用代理对象方法
- 匹配注解 / 切点
- 执行 advice
- 再调用目标方法

最本质的一句是：

`Aspect 不是注解的同义词，而是 Spring 中解释某些注解并织入横切逻辑的一种机制`

---

## 8. Spring 的核心心智模型

### 8.1 Spring 首先是容器

Spring 的核心身份不是“注解库”，而是：

`应用容器`

它负责：

- 创建对象
- 注入依赖
- 管理对象关系
- 管理生命周期
- 增强对象能力

所以 Spring 管的不是一个个类，而是：

`Bean 依赖图`

### 8.2 BeanDefinition 是“对象说明书”

Spring 启动时，并不是一上来就创建所有对象。

它通常先构建 `BeanDefinition`，可以理解成：

- 类名
- 作用域
- 依赖
- 初始化方式
- 销毁方式

这些信息先被收集起来，再决定如何真正实例化 Bean。

所以：

- `BeanDefinition` 是定义
- `Bean` 是实例

### 8.3 生命周期是 Spring 的主线

典型 Bean 生命周期：

- 读取配置
- 注册 BeanDefinition
- 执行 `BeanFactoryPostProcessor`
- 实例化 Bean
- 属性注入
- `Aware` 回调
- `BeanPostProcessor` 前置处理
- 初始化
- `BeanPostProcessor` 后置处理
- Bean 可用
- 容器关闭时销毁

关键理解：

`Spring 的很多问题，本质上都是“发生在生命周期的哪个阶段”`

### 8.4 注解在 Spring 里只是输入

Spring 里很多注解本质上都是元数据：

- `@Component`
- `@Autowired`
- `@Transactional`
- `@Aspect`
- `@RequestMapping`

它们不是执行器。  
真正执行的是：

- 扫描器
- 后处理器
- 代理器
- 映射器
- 拦截器

所以：

`Spring = 元数据解释系统`

## 9. Spring AOP 的心智模型

### 9.1 AOP 的本质

AOP 的本质不是“修改代码”，而是：

`改变调用路径，在方法调用前后织入统一逻辑`

典型场景：

- 日志
- 权限
- 审计
- 事务
- 缓存
- 限流

### 9.2 Spring AOP 常依赖代理

你以为拿到的是：

- `UserService`

实际上很多时候拿到的是：

- `Proxy(UserService)`

调用链是：

- 调用方
- 代理对象
- advice / interceptor
- 目标方法

所以：

`Spring AOP 改的是调用路径，不是方法本体`

### 9.3 为什么 `@Transactional` 适合 AOP

事务逻辑天然是横切逻辑：

- 开始事务
- 调用业务
- 成功提交
- 失败回滚

如果每个方法手写，会大量重复。

所以 Spring 用：

- `@Transactional` 声明需求
- 事务拦截器统一处理

统一模型：

- `@Transactional`：声明
- 事务拦截器：解释器
- 代理：执行载体

### 9.4 同类内部调用为什么常失效

这是 Spring AOP 最经典的问题之一。

原因：

- 外部调用会经过代理
- 同类内部 `this.xxx()` 往往不会经过代理
- 没进代理就没进增强链

所以：

`AOP 是否生效，关键看调用有没有经过代理`

---

## 10. Spring Boot 的心智模型

Spring Boot 不是替代 Spring，而是：

`约定化装配 Spring`

它主要做这些事：

- 自动配置
- 条件装配
- starter 依赖收敛
- 默认工程结构和基础设施配置

核心思想：

`如果满足条件，就自动注册合适的 Bean`

这就是各种 `@Conditional...` 的本质。

---

## 11. 如何整体理解 Spring

可以把 Spring 压成 6 个关键词：

- `Container`
  - 容器
- `Bean Graph`
  - 对象关系图
- `Lifecycle`
  - 生命周期
- `Metadata`
  - 元数据
- `Proxy`
  - 代理与增强
- `Extension Points`
  - 扩展点

只要这 6 个词在脑子里清楚，Spring 就不会散。

---

## 12. 一张主线图应该怎么读

Spring 心智图的推荐阅读顺序：

### 12.1 顶层目标

- Spring 是应用容器

### 12.2 核心内核

- `ApplicationContext`
- `BeanDefinition`
- Bean 生命周期

### 12.3 运行增强

- 元数据
- 扩展点
- AOP / 代理
- 事务

### 12.4 上层子系统

- Boot
- Web
- Data
- Security

阅读时持续问 4 个问题：

- 注册了什么 Bean
- 在生命周期哪个阶段介入
- 有没有创建代理
- 元数据由谁解释

这是理解 Spring 最实用的分析框架。

## 13. 常见误区

### 13.1 误区一：注解自己会执行逻辑

错误。  
注解只是元数据，真正执行的是解释器。

### 13.2 误区二：Spring 就是一堆注解

错误。  
Spring 本质是容器和扩展框架，注解只是输入方式之一。

### 13.3 误区三：AOP 是修改原方法

错误。  
Spring AOP 常见方式是代理，不是直接改原方法。

### 13.4 误区四：只会 `@Autowired`、`@Transactional` 就算懂 Spring

错误。  
真正理解 Spring，要理解：

- 容器
- 生命周期
- BeanDefinition
- 后处理器
- 代理链

## 15. 复习卡片

### 卡片 1

问：注解为什么不会自己生效？  
答：因为注解只是元数据，必须由编译器、反射器、代理器或框架解释并执行。

### 卡片 2

问：`@Aspect` 和业务注解的关系是什么？  
答：业务注解负责声明需求，`@Aspect` 负责匹配这些标记并织入横切逻辑。

### 卡片 3

问：Spring AOP 的本质是什么？  
答：通过代理对象改变调用路径，在方法调用前后统一插入增强逻辑。

### 卡片 4

问：Spring 最核心的身份是什么？  
答：应用容器，负责管理 Bean 图、生命周期、元数据解释和基础设施织入。

### 卡片 5

问：为什么同类内部调用常导致 `@Transactional` 失效？  
答：因为调用没有经过 Spring 代理对象，事务拦截器没有机会介入。

---

## 16. 一句话总结

`理解 Spring，不是记住多少注解，而是理解容器如何构建对象图、如何在生命周期中解释元数据、如何通过代理织入横切能力。`
